There is a version of "AI-assisted development" that looks like an IDE plugin with autocomplete. That version is already here and already commoditised. The more consequential version — the one engineering organisations are currently building — looks like a local agent process that understands your service's conventions, respects your platform's security tier, talks to a verification loop before raising a PR, and writes every action to a central audit log the developer cannot modify.

These are not the same thing. The first improves individual keystrokes. The second changes who can make what kind of change, at what scale, with what level of confidence.

## What Actually Changed

The developer desktop used to be three systems: IDE, terminal, CI. The IDE is where you write code. The terminal is where you run it. CI is where it gets validated before merging.

The modern developer desktop for agent-assisted work adds four more systems. A local **skills library** that encodes your organisation's conventions, patterns, and gotchas in a form the agent can read before generating code. An **agent runner** configured to your service's risk tier and locked to the repositories and task types your platform team permits. A **verification loop** — the combination of a deterministic verifier that runs build, test, and lint, and an LLM Judge that evaluates the diff semantically before a PR is opened. And a **platform connection** that checks the kill switch, syncs configuration updates, and streams every session event to a central audit log.

[Spotify's Honk engineering series](https://engineering.atspotify.com/2025/11/spotifys-background-coding-agent-part-1) — three posts documenting their background coding agent that has merged over 1,500 PRs into production — makes this architecture concrete. They describe exactly this structure: a verification MCP that abstracts build complexity, an LLM Judge that vetoes roughly a quarter of all sessions, a CLI that wraps the agent and handles audit logging, and a deliberate decision to keep the agent's tool surface minimal.

## Zone 1 — The Skills Library

Skills are the mechanism for encoding institutional knowledge. Not documentation. Not a wiki page. Versioned folders — checked into a platform-managed registry — that contain instruction files, concrete before/after code examples from your actual codebase, gotcha files covering what the agent gets wrong by default, and executable scripts the agent can run to verify its own output.

[Anthropic's Claude Code team describes this directly](https://www.anthropic.com/claude-code): a common misconception is that skills are "just markdown files." The more interesting skills are folders with scripts, assets, and data that the agent can discover and execute. The description of a skill is written for the model — it is the trigger that causes the agent to load the skill when it is relevant.

For an enterprise estate with domain services, aggregate services, JBPM orchestration, Cassandra and Oracle persistence, and multiple SOE frontend channels, the skills library contains:

- **`cassandra-patterns/`** — CQL idioms, keyspace naming conventions, and the gotcha the agent always makes: using `session.execute()` with a string literal instead of a `PreparedStatement`. Includes a verification script that flags this pattern on any file the agent edits.
- **`jbpm-handler-stubs/`** — Process handler templates, `@WorkItemHandler` annotation patterns, and an explicit instruction: process definition files (`*.bpmn`, `*.drl`) are never to be touched.
- **`oracle-sp-conventions/`** — Stored procedure call patterns, JDBC parameter binding, schema name casing conventions your codebase uses that the agent will get wrong without this skill.
- **`product-verification/`** — The most important skill. A script that runs build, test, and lint for your service archetype. [Spotify found this worth a full engineer-week of investment per agent type](https://engineering.atspotify.com/2025/11/context-engineering-background-coding-agents-part-2). Without it, the agent has no feedback loop.
- **`channel-patterns/`** — Per SOE channel component and handler patterns, plus explicit lists of what is blocked for each channel.

The platform team publishes the skills library. Developers pull it. This is what makes agent knowledge consistent across teams without requiring every developer to understand every layer of the stack.

## Zone 2 — The Agent Configuration Layer

The developer does not write this file. They run `./setup.sh --service customer-profile-service` and the platform's setup script fetches the right configuration from the central registry, validates it against the platform baseline, and writes it as a local `agent_config.yaml`. This file locks the agent to that developer's service context.

The configuration defines: which repositories the agent can touch (a named list, no wildcards), which task types are permitted, the change budget (files per session, enforced as a hard error), the deny list paths from the central registry, which LLM family to use as judge (must differ from the coding agent's family), and the kill switch endpoint checked at session start.

The platform team manages risk tiers that determine the defaults. Higher-risk tiers (IVR, payments, authentication, POS transaction handlers) get tighter change budgets and higher judge confidence requirements. Lower-risk tiers (domain services, digital frontend) get more headroom. The developer inherits the tier their service is registered under.

At every session start, the runner compares the local config hash against the platform registry. If the platform team pushed a new deny list entry, the developer's next session picks it up automatically. Stale config means the session is blocked until the developer re-runs setup.

## Zone 3 — The Verification Loop

This is the most important zone — and the most expensive to build correctly.

The deterministic verifier runs build, test, and lint. It auto-detects the build system based on repository contents — Maven `pom.xml` present? Spring Boot? Oracle JDBC? — and runs the right verifier without the agent needing to understand the specifics. [Spotify encodes this as a Model Context Protocol tool](https://engineering.atspotify.com/2025/12/feedback-loops-background-coding-agents-part-3). The verifier abstracts away dozens of different build configurations and returns clean, summarised output to the agent. The agent knows only that it can call `verify()` and that verify must pass. It does not know what verify does.

On top of the deterministic verifier, the LLM Judge provides semantic review. The judge uses a different model family from the coding agent — same family means shared blind spots. The judge reviews the diff against the original task description and returns a structured JSON verdict: `{ verdict, confidence, scope_violations, test_integrity_issues, reasoning }`. Low-confidence approvals are routed to a human queue, not to merge.

[Spotify's judge vetoes roughly 25% of all sessions](https://engineering.atspotify.com/2025/12/feedback-loops-background-coding-agents-part-3). The agent corrects itself about half the time when vetoed. The remaining sessions go to a dead-letter queue for human review.

Gate 3 runs deterministic diff checks before the judge: secret scanning, test count delta (agent's changes cannot reduce test count), and a prohibited pattern scanner that flags `@Ignore`, `@Disabled`, `.skip()`, `eslint-disable`, `maven.test.skip=true`. These are the patterns an agent uses to make failing code appear to pass verification.

## Zone 4 — The Platform Connection

The developer's desktop is a node in the platform, not an island.

At session start, the runner checks the kill switch endpoint — a Redis-backed flag activatable in under thirty seconds. When triggered, all new sessions across all developer desktops are refused. In-flight sessions complete their current verification iteration but do not open PRs.

Per-domain circuit breakers provide surgical intervention. The runner monitors the judge veto rate per domain across a rolling session window. If the rate exceeds the tier's configured threshold, the circuit breaker trips automatically, the domain is paused, and an alert is sent. Human intervention is required to re-enable.

Every session event streams to a central append-only audit log: tool call inputs and outputs, LLM prompts, judge verdict, MR link. The log is write-only from the agent's perspective. Given a session ID, the platform team can reconstruct exactly what the agent read, what it generated, why the judge approved or rejected it.

The platform team's instrument is the weekly compliance dashboard: sessions per team, judge approval rate by tier, most common Gate 3 rejection reasons, services with elevated veto rates, desktop runners on stale configuration. They review the pattern, not individual sessions.

## What a Session Looks Like End to End

A developer runs: `agent run --task "Update spring-boot-starter-parent from 3.1.0 to 3.2.0"`

The runner checks the kill switch. It verifies config freshness. It validates the task type against the permitted list. It loads the relevant skills. It registers the session and captures a baseline snapshot — git SHA, test count, coverage percentage.

The agent runs in a sandboxed container with the deny list enforced at the tool level. It modifies `pom.xml`. It calls `verify()`. The verifier returns: tests pass, test count increased. Gate 3 runs: no secrets, no prohibited patterns, test count delta positive. Gate 4: judge (different model family) returns APPROVE with high confidence. The orchestrator creates the MR with mandatory labels: `ai-generated`, `agent-session:sess_abc123`, `tier3`.

The developer sees: MR opened, judge approved, session logged. They review the diff and merge. They did not configure a single guardrail. The platform configured all of them.

## What Is Not on the Desktop — Intentionally

[The Spotify team made a deliberate choice about tool minimalism](https://engineering.atspotify.com/2025/11/context-engineering-background-coding-agents-part-2): no code search tool, no documentation fetcher exposed to the agent at runtime. Context goes in up front via the skills library. The agent's runtime tool surface is minimal: a constrained `edit_file`, a `read_file` with content sanitization, a `verify()` MCP, and a Git tool that never allows push or origin changes.

This is not a limitation. It is a design decision. More tools mean more dimensions of unpredictable failure. An agent with access to code search across the organisation's repositories is an agent that can read content from repositories it was not configured to touch. The minimal tool surface is part of the security model.

What the agent does not know from the skill, it does not know.

## What This Means for the Platform Team

The platform team's role changes in this model. They are no longer gatekeepers who review individual PRs. They are the publishers of the infrastructure that makes all agent sessions safe by default.

They publish the skills library. They update deny lists when new vulnerability classes emerge. They review the weekly compliance dashboard. They handle circuit breaker escalations. They manage tier assignments as services are added or retired. They own the judge configuration.

The developer's desktop is safe not because the developer configured it correctly. It is safe because the platform team published the configuration the desktop runner enforces.
