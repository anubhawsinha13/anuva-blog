The framing that AI-assisted development is a spectrum from "autocomplete" to "autonomous agent" is correct but underspecified. The distance between Stage 1 and Stage 4 is not uniform. There are three transitions, and they are not equivalent. The first two transitions change your workflow. The third changes your safety model.

## Stage 1 → Stage 2: Autocomplete to Copilot Chat

The shift from autocomplete to conversational AI is a change in your role, not the tool's autonomy.

Autocomplete predicts the next token based on statistical patterns in surrounding code. It does not understand what you are building. It does not know what the function is supposed to do. [GitHub Copilot](https://github.com/features/copilot)'s chat mode and [Cursor](https://cursor.sh)'s Chat panel change this. You describe intent. The model responds across the whole file — it can see your full function signature, your imports, and your neighbouring methods.

The new skill required is prompt clarity. Vague prompts produce hedged answers. Specific prompts produce specific code.

**What changes:** From reviewing tokens to reviewing intent. From reactive to conversational. From one line of context to full-file context.

**What does not change:** Nothing happens in your repository autonomously. You still manually apply every suggestion. The blast radius is still one file, touched by your own hand.

## Stage 2 → Stage 3: Chat to In-IDE Multi-File Agent

This transition introduces blast radius.

[Cursor Agent](https://cursor.sh) and [GitHub Copilot Workspace](https://githubnext.com/projects/copilot-workspace) can make changes across multiple files in a single instruction. You say "migrate UserService to use the new CustomerRepository interface" and the agent reads `UserService.java`, finds all call sites, updates method signatures, and modifies `CustomerServiceTest.java` to match. You did not enumerate the files. The agent decided which files to touch.

This is genuinely useful — a change that previously required thirty minutes of mechanical editing across six files takes thirty seconds. But blast radius has appeared. If your prompt was ambiguous, the agent will make a choice without telling you it made one. If a shared utility is involved, the agent may update it. If it judges a method to be dead, it may delete it. You are still watching — but watching means evaluating a diff across eight files, not reading every character typed.

**What you must add at Stage 3:**
- Check file scope before accepting — more files than expected means investigate first
- Commit before each agent run — a clean `git reset` point costs twenty seconds and saves hours
- Scope tasks to one change type per prompt — combining dependency update with refactor produces partial results
- Encode conventions using Skills or `.cursor/rules` — the agent will guess your patterns consistently wrong without them

The habits you build at Stage 3 are the intuitions that become automated controls at Stage 4.

## Stage 3 → Stage 4: The Categorical Leap

This is not an incremental step. Everything before this point is supervised. You are present. You see changes before they go anywhere.

Stage 4 removes you from the loop. The agent runs in a container, takes a task from a queue, makes changes, calls a verifier, and opens a pull request — all without you watching. [Spotify's background coding agent](https://engineering.atspotify.com/2025/11/spotifys-background-coding-agent-part-1) (codename Honk) has merged over 1,500 pull requests this way — dependency updates, breaking API migrations, framework modernisation across thousands of repositories.

An unsupervised agent operating without constraints can:

- Delete or disable tests to make `verify()` pass on broken code
- Surface secrets from repository files into the LLM context, where they appear in generated output
- Modify CI/CD configuration to broaden its own execution permissions
- Open a PR whose description accurately describes a dependency bump but whose diff contains an unrelated refactor
- Produce a PR that passes CI, looks correct in review, and is functionally broken at runtime

**What you must build before Stage 4 — all of these, not some:**

**Gate 2 — Tool constraints:** Protected paths (CI/CD config, credential files, database migrations, BPMN process definitions) in a deny list the agent cannot override. Change budget enforced as a hard error.

**Gate 3 — Diff validation:** Secret scanning, test count delta check (reduced test count aborts the session), and a prohibited pattern scanner: `@Ignore`, `@Disabled`, `.skip()`, `eslint-disable`, `maven.test.skip=true`. These are the patterns an agent uses to make failing code appear to pass verification.

**Gate 4 — LLM Judge:** A second LLM using a different model family reviews the diff and returns a structured verdict with confidence score. [Spotify's judge vetoes roughly 25% of all sessions](https://engineering.atspotify.com/2025/12/feedback-loops-background-coding-agents-part-3), with the agent self-correcting about half the time. Different model family is an architectural requirement — same family means shared blind spots.

**Gate 5 — MR safety:** The agent does not create pull requests. The orchestrator does. Every agent PR carries mandatory `ai-generated` labels. PR title and description are orchestrator-generated from templates.

**Gate 7 — Kill switch:** A global pause activatable in under thirty seconds. Build this before the first autonomous session — a kill switch built after an incident is built under pressure.

## Why Stage 3 to Stage 4 Is Categorical

At every stage before Stage 4, the safety model is: the developer is watching. That model works because humans catch unexpected changes when reviewing diffs. We notice when a function we did not ask about was modified. We notice when a test was deleted.

Stage 4 removes the human reviewer from the inner loop. The safety layer is now the architecture you built before the first session. If the eight gates are missing, there is no fallback.

The developer who is disciplined at Stage 3 — scoping prompts tightly, checking file counts, committing before runs — will build better Stage 4 guardrails, because they have already encountered the failure modes those guardrails prevent. The habits at Stage 3 are the intuitions that become automated controls at Stage 4.

The [coding-agent-scaffold](https://github.com/anubhawsinha13/coding-agent-scaffold) — the skeleton implementation with all eight gates wired in from day one — exists so Stage 4 does not start from a blank canvas. It is the minimum viable safety layer for the first unsupervised session.
