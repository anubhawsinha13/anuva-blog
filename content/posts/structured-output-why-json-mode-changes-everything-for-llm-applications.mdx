An LLM extracts a contact from an email and returns: "The person's name is Sarah Chen. She works as a senior engineer at Datadog." That's useful to a human reading it. It's useless to code that needs `contact.name` and `contact.email`.

This is the gap between **LLM output for humans** and **LLM output for systems**. Structured output — forcing the model to respond in JSON, XML, or another parseable format — closes it.

## The Parsing Problem

Free text from an LLM is non-deterministic in format. Ask for "the key findings" twice and you might get:

- A numbered list the first time
- Bullet points the second time
- A paragraph the third time

All contain the same information. None can be reliably parsed by code. You end up writing fragile regex patterns that break when the model slightly changes its phrasing — which it will, because every response is a fresh sample from a probability distribution.

## How Structured Output Works

Modern LLM APIs offer several mechanisms for structured output:

**JSON mode**: The API constrains the model to produce valid JSON. The model's logits (token probabilities) are masked so that only tokens that would produce valid JSON structure can be selected.

**Function calling / tool use**: The developer defines a schema (function parameters), and the model fills in the values. This guarantees not just valid JSON but valid structure — the correct keys are always present.

**Schema enforcement**: Some APIs accept a JSON Schema definition and constrain the output to match it exactly. Missing fields, wrong types, and extra fields are impossible.

```
// Without structured output
const text = await llm.generate("Extract contact info from...");
const name = text.match(/name is (.+?)\./)?.[1]; // fragile regex

// With structured output
const contact = await llm.generate("Extract contact info from...", {
  response_format: { type: "json_object" }
});
const { name, email, company } = JSON.parse(contact); // deterministic keys
```

## When Structured Output Is Essential

**Pipeline integration**: When LLM output feeds into code — routing tickets, populating databases, triggering workflows — free text requires a parsing layer that is inherently brittle.

**Multi-agent systems**: When one LLM's output is another LLM's input, structured formats prevent information loss and format confusion across agent boundaries.

**CI/CD automation**: Code review findings need severity levels and line numbers in parseable fields for automated PR blocking. "I found a critical issue" in prose can't trigger a pipeline gate.

**Evaluation and testing**: Structured output enables automated evaluation — you can programmatically check if the model returned the expected classification, extracted the right fields, or identified the correct issues.

## The Trade-Off

Structured output adds constraints that can slightly reduce the model's expressiveness:

- **Nuance compression**: "This is mostly a billing issue but also touches account access" becomes `"categories": ["billing", "account-access"]` — the nuance of "mostly" is lost
- **Token overhead**: JSON syntax (`{`, `"`, `:`, `,`) consumes tokens that could hold content
- **Schema rigidity**: If the model encounters information that doesn't fit the schema, it has no way to flag it

The solution is to include an `"additional_notes"` or `"flags"` field in your schema for anything that doesn't fit the structured fields. This gives the model an escape valve while keeping the primary output parseable.

## Key Takeaways

- Free text is for humans; structured output is for systems
- JSON mode constrains the model to produce valid, parseable output with deterministic keys
- Structured output eliminates the fragile regex parsing layer that breaks when phrasing changes
- Essential for pipeline integration, multi-agent systems, CI/CD automation, and evaluation
- Include an escape-valve field in your schema for information that doesn't fit the structured fields
- The information content is the same — structured output changes the format, not the quality

When your LLM output feeds into code, JSON mode is not optional. It is the difference between a demo and a production system.
